# PlantX Kit 平台架构设计

> 版本：v1.0  
> 日期：2026-06-17  
> 定位：面向业务开发者的 Kit 平台，基础设施集中收敛，业务层零感知。

---

## 一、设计目标

| 目标 | 说明 |
|---|---|
| **契约驱动（Contract-First）** | 一切从 `proto` 和 `sqlc` 出发，前后端、SDK、UI 共享同一份契约 |
| **领域驱动设计（DDD）** | 业务代码聚焦领域模型，平台能力通过 Kit 运行时注入 |
| **业务开发零关注基础设施** | 认证、鉴权、租户、日志、事件、路由、部署等由平台团队负责 |
| **实现可替换** | 所有基础设施通过抽象接口接入，不绑定具体实现（MaxKey/Keycloak、OPA/Casbin、K8s/Docker） |
| **支持多部署形态** | 既能在 K8s 运行，也能在 Docker Compose 或裸机环境运行 |

---

## 二、核心设计原则

### 2.1 契约约束链

```text
proto + sqlc  >  sdk  >  kit  >  ui
```

- **proto**：服务间接口、前后端接口、事件定义的源头
- **sqlc**：数据访问层契约，数据库 schema 与查询语句即契约
- **sdk-api**：由 proto 生成的 API 调用层
- **sdk-kit**：基于 sdk-api 封装的业务 hooks / utils
- **ui**：只依赖 sdk-kit 和 kit-ui，不直接调用 API

### 2.2 抽象接口原则

平台不强制任何具体实现，所有基础设施通过 Go interface / TypeScript abstraction 接入：

```go
// 认证抽象
type Authenticator interface {
    Authenticate(ctx context.Context, token string) (*UserInfo, error)
}

// 鉴权抽象
type Authorizer interface {
    Authorize(ctx context.Context, req AuthorizationRequest) (Decision, error)
}

// 事件总线抽象
type EventBus interface {
    Publish(ctx context.Context, event domain.Event) error
    Subscribe(subject string, handler EventHandler) error
}

// 租户解析抽象
type TenantResolver interface {
    Resolve(ctx context.Context, user UserInfo) (TenantInfo, error)
}
```

默认实现：

| 能力 | 默认实现 | 可替换实现 |
|---|---|---|
| 认证 | MaxKey（OIDC/JWT） | Keycloak、自研 IAM、LDAP |
| 鉴权 | OPA（Rego） | Casbin、自研 RBAC |
| 事件总线 | NATS JetStream | Kafka、RabbitMQ、Redis Pub/Sub |
| 日志 | OpenTelemetry + Loki | ELK、SLS、Fluentd |
| 追踪 | OpenTelemetry + Jaeger/Tempo | SkyWalking、Zipkin |
| 配置中心 | 环境变量 + 文件 | Apollo、Nacos、Consul |
| 服务发现 | K8s DNS / Docker Compose DNS | Consul、Nacos、静态配置 |
| 部署 | K8s Helm / Docker Compose | systemd、二进制 + Nginx |

### 2.3 横切关注点收敛

业务服务中不允许直接写：

- JWT 解析
- OPA/Casbin 调用
- 租户 ID 手动注入
- Kafka/NATS 原始 SDK 调用
- 链路追踪手动埋点

以上全部由 Kit 运行时通过拦截器、上下文、代码生成完成。

## 四、部署策略：K8s + 非 K8s 双模式

### 4.1 设计原则

- 所有服务必须**容器化**（Dockerfile）
- 服务启动只依赖**环境变量 + 配置文件**
- 服务发现、配置中心通过抽象接口接入，不同环境提供不同实现
- 网关层在非 K8s 环境下使用 Nginx / APISIX / Traefik 替代 Ingress

### 4.2 模式一：Kubernetes + Helm（推荐生产环境）

```text
┌─────────────────────────────────────┐
│  Ingress / APISIX / Istio Gateway   │
└───────────────┬─────────────────────┘
                │
        ┌───────▼────────┐
        │   PlantX Helm  │
        │  ├─ Gateway    │
        │  ├─ 业务服务    │
        │  ├─ MaxKey     │
        │  ├─ OPA        │
        │  ├─ PostgreSQL │
        │  ├─ NATS       │
        │  └─ Loki       │
        └────────────────┘
```

- 服务发现：K8s DNS + Service
- 配置管理：ConfigMap / Secret
- 服务网格：可选 Istio/Linkerd
- 部署单元：Deployment + Service + Ingress

### 4.3 模式二：Docker Compose（中小规模 / 私有化交付）

```yaml
# deployments/docker-compose/docker-compose.yml 示意
services:
  gateway:
    image: plantx/gateway:latest
    ports:
      - "80:80"
  maxkey:
    image: maxkey/maxkey:latest
  opa:
    image: openpolicyagent/opa:latest
  order-service:
    image: plantx/order-service:latest
    environment:
      - KIT_DB_DSN=postgres://...
      - KIT_AUTH_PROVIDER=maxkey
      - KIT_AUTHZ_PROVIDER=opa
      - KIT_EVENT_BUS=nats
  postgres:
    image: postgres:16
  nats:
    image: nats:latest
```

- 服务发现：Docker Compose 内置 DNS（服务名即 host）
- 配置管理：环境变量 + 挂载配置文件
- 适合：私有化部署、POC、中小型客户

### 4.4 模式三：二进制 + 外部中间件（裸机 / 虚拟机）

```text
Nginx / APISIX
    ├── order-service (systemd / supervisor)
    ├── product-service
    └── MaxKey / OPA / PostgreSQL / NATS
```

- 每个服务编译为二进制，通过 systemd 管理
- 中间件使用外置集群或云托管（RDS、云 Kafka 等）
- 适合：信创环境、客户已有中间件、需要简化容器运行时

### 4.5 部署抽象矩阵

| 能力 | K8s 实现 | 非 K8s 实现 |
|---|---|---|
| 网关 | Ingress / Istio Gateway / APISIX | Nginx / APISIX / Traefik |
| 服务发现 | K8s DNS + Service | Docker DNS / Consul / 静态配置 |
| 负载均衡 | K8s Service | Nginx Upstream / Consul |
| 配置中心 | ConfigMap / Secret | 环境变量 / 文件 / Nacos |
| 日志收集 | DaemonSet Fluent-Bit + Loki | Docker Logging Driver / Filebeat |
| 服务治理 | Istio / Linkerd | APISIX / 自研 Gateway |

## 五、认证与鉴权设计

### 5.1 认证：MaxKey（OIDC）

MaxKey 作为默认认证中心，对外提供标准 OIDC / OAuth2。

**Kit 抽象接口：**

```go
package auth

type UserInfo struct {
    ID          string
    TenantID    string
    Username    string
    DisplayName string
    Email       string
    Roles       []string
    Permissions []string
    Claims      map[string]string
}

type Authenticator interface {
    Authenticate(ctx context.Context, token string) (*UserInfo, error)
}
```

**默认实现：`maxkey.Authenticator`**

```go
// 通过 OIDC UserInfo Endpoint 或本地 JWT 公钥验证
type authenticator struct {
    issuer   string
    jwksURL  string
    clientID string
}

func (a *authenticator) Authenticate(ctx context.Context, token string) (*auth.UserInfo, error) {
    // 1. 解析 JWT
    // 2. 验证签名（本地 JWKS 或 MaxKey introspection）
    // 3. 提取 user / tenant / roles / permissions
    // 4. 返回 UserInfo
}
```

**切换方式：**

```yaml
kit:
  auth:
    provider: maxkey  # 可改为 keycloak / custom
    maxkey:
      issuer: https://sso.plantx.local
      client_id: plantx-portal
```

### 5.2 鉴权：OPA（默认）

**Kit 抽象接口：**

```go
package authz

type Action struct {
    Service    string
    Resource   string
    Operation  string
}

type AuthorizationRequest struct {
    User            auth.UserInfo
    Action          Action
    ResourceContext map[string]any
}

type Authorizer interface {
    Authorize(ctx context.Context, req AuthorizationRequest) (Decision, error)
}
```

**默认实现：`opa.Authorizer`**

- OPA 以 Sidecar 或独立服务部署
- Kit 拦截器在每次 RPC/HTTP 调用前自动调用 OPA
- 业务服务通过 proto annotation 声明策略

```protobuf
service OrderService {
  rpc CreateOrder(CreateOrderRequest) returns (Order) {
    option (plantx.authz.action) = {
      service: "order"
      resource: "order"
      operation: "create"
    };
  }
}
```

**切换方式：**

```yaml
kit:
  authz:
    provider: opa  # 可改为 casbin / custom
    opa:
      endpoint: http://localhost:8181
      policy_bundle: plantx/authz
```

### 5.3 多租户：默认行级隔离

**租户上下文：**

```go
package kitctx

func GetTenant(ctx context.Context) TenantInfo
func GetUser(ctx context.Context) auth.UserInfo
func WithTenant(ctx context.Context, t TenantInfo) context.Context
```

**租户 ID 来源：**

1. Gateway 从 MaxKey JWT 中解析 `tenant_id`
2. 通过 gRPC metadata / HTTP header 注入：`x-tenant-id`
3. Kit 拦截器自动注入 `context`

**数据层自动隔离：**

- 所有业务表必须包含 `tenant_id` 字段
- Kit sqlc 插件自动为查询追加 `WHERE tenant_id = $N`
- 业务 SQL 写：

```sql
-- name: ListOrders :many
SELECT * FROM orders WHERE status = $1;
```

- 生成后实际执行：

```sql
SELECT * FROM orders WHERE tenant_id = $2 AND status = $1;
```

## 六、后端架构：DDD + Contract-First

### 6.1 DDD 分层

```text
services/order/
├── api/
│   └── order.proto              # 接口契约（proto）
├── internal/
│   ├── domain/                  # 领域层
│   │   ├── order.go             # 聚合根
│   │   ├── order_repository.go  # 仓储接口
│   │   └── order_event.go       # 领域事件
│   ├── app/
│   │   └── order_service.go     # 应用服务（编排领域对象）
│   ├── infra/
│   │   ├── sqlc/
│   │   │   └── queries.sql      # sqlc 查询契约
│   │   │   └── schema.sql       # 表结构（或迁移文件）
│   │   └── repo/
│   │       └── order_repo.go    # 仓储实现
│   └── interfaces/
│       └── grpc/
│           └── order_handler.go # gRPC 服务实现（通常由 kit 生成骨架）
└── cmd/
    └── main.go                  # 启动入口（kit 框架启动）
```

### 6.2 各层职责

| 层级 | 职责 | 依赖规则 |
|---|---|---|
| **domain** | 实体、值对象、领域事件、仓储接口、领域服务 | 不依赖任何外部框架 |
| **app** | 用例编排、事务边界、DTO 转换 | 可依赖 domain |
| **infra** | 仓储实现、sqlc、外部客户端 | 可依赖 domain + kit |
| **interfaces** | gRPC/HTTP handler | 可依赖 app + domain + kit |

### 6.3 proto 设计规范

```protobuf
syntax = "proto3";
package plantx.order.v1;

import "plantx/kit/authz.proto";
import "plantx/kit/context.proto";

service OrderService {
  rpc CreateOrder(CreateOrderRequest) returns (Order) {
    option (plantx.authz.action) = { service: "order" resource: "order" operation: "create" };
  }

  rpc ListOrders(ListOrdersRequest) returns (OrderList) {
    option (plantx.authz.action) = { service: "order" resource: "order" operation: "list" };
  }
}

message Order {
  string id = 1;
  string tenant_id = 2;  // 租户字段，kit 自动处理
  string status = 3;
  int64 created_at = 4;
}
```

### 6.4 sqlc 设计规范

```sql
-- migrations/001_init.up.sql
CREATE TABLE orders (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   TEXT NOT NULL,
    status      TEXT NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_orders_tenant_status ON orders(tenant_id, status);
```

```sql
-- internal/infra/sqlc/queries.sql
-- name: CreateOrder :one
INSERT INTO orders (tenant_id, status)
VALUES ($1, $2)
RETURNING *;

-- name: ListOrdersByStatus :many
SELECT * FROM orders
WHERE status = $1
ORDER BY created_at DESC;
```

> Kit 的 sqlc 插件会自动把 `tenant_id` 注入到需要它的查询中。

## 七、前端架构：qiankun + 三层包

### 7.1 包约束链

```text
proto  >  {service}-sdk-api  >  {service}-sdk-kit  >  {service}-ui
           ↑ kit-sdk-api      ↑ kit-sdk-kit         ↑ kit-ui
```

| 包名 | 来源 | 职责 | 禁止依赖 |
|---|---|---|---|
| `kit-sdk-api` | 平台维护 | HTTP 请求封装、错误处理、认证 header 注入 | 不依赖业务包 |
| `kit-sdk-kit` | 平台维护 | 通用 React hooks、权限 hooks、租户上下文 | 不依赖业务包 |
| `kit-ui` | 平台维护 | 通用 AntD 组件、布局、主题 | 不依赖业务包 |
| `{service}-sdk-api` | 由 proto 生成 | 该服务的 API 调用 | 不依赖 kit-sdk-kit/ui |
| `{service}-sdk-kit` | 业务编写 | 该服务的业务 hooks、状态管理 | 仅依赖 {service}-sdk-api + kit-sdk-* |
| `{service}-ui` | 业务编写 | 该服务的页面与组件 | 仅依赖 {service}-sdk-kit + kit-ui |

### 7.2 微前端主应用职责

- 统一对接 MaxKey OIDC 登录
- 获取用户/租户/权限信息
- 根据权限动态渲染菜单
- 通过 qiankun props 向子应用注入：

```ts
interface KitMicroAppProps {
  user: KitUser;
  tenant: KitTenant;
  permissions: string[];
  eventBus: KitEventBus;
  apiClient: KitApiClient;
}
```

### 7.3 子应用开发规范

```tsx
// services/order/web/src/pages/OrderList.tsx
import { useOrders } from '@plantx/order-sdk-kit';
import { useKitPermission } from '@plantx/kit-sdk-kit';

export default function OrderList() {
  const { data, loading } = useOrders();
  const { hasPermission } = useKitPermission();

  return (
    <PageContainer>
      {hasPermission('order:create') && <Button type="primary">新建订单</Button>}
      <Table dataSource={data} loading={loading} />
    </PageContainer>
  );
}
```

子应用不直接调用后端，也不直接对接 OIDC。

## 八、Kit 运行时核心能力

### 8.1 Kit-Go 模块划分

```text
kit/kit-go/
├── auth/           # 认证抽象 + MaxKey 实现
├── authz/          # 鉴权抽象 + OPA 实现
├── tenant/         # 租户上下文与解析
├── context/        # 通用上下文（user/tenant/trace）
├── event/          # 事件总线抽象 + NATS 实现
├── db/             # sqlc 封装、连接池、事务
├── log/            # 结构化日志（OpenTelemetry 标准）
├── trace/          # 分布式追踪
├── server/         # gRPC/HTTP 服务器启动、拦截器链
├── config/         # 配置加载抽象
├── discovery/      # 服务发现抽象
├── errors/         # 统一错误码
└── proto/          # kit 通用 proto（authz/context/event）
```

### 8.2 拦截器链顺序

```text
请求进入
  → Recovery（panic 恢复）
  → Trace（生成/继承 trace_id）
  → Logging（请求日志）
  → Authenticate（JWT 验证）
  → Tenant Resolve（租户解析）
  → Authorize（OPA 鉴权）
  → Metrics（耗时统计）
  → 业务 Handler
```

### 8.3 事件总线抽象

```go
// 业务代码
kit event.Publish(ctx, &orderpb.OrderCreatedEvent{
    OrderId: order.ID,
})

// Kit 自动完成：
// 1. 序列化事件
// 2. 注入 trace/tenant/user 上下文
// 3. 发送到 NATS/Kafka
// 4. 失败落库 + 重试
```

## 十、可替换性设计检查表

| 基础设施 | 抽象接口位置 | 默认实现 | 替换成本 |
|---|---|---|---|
| 认证 | `kit-go/auth.Authenticator` | `auth/maxkey` | 改配置 + 新增 provider 包 |
| 鉴权 | `kit-go/authz.Authorizer` | `authz/opa` | 改配置 + 新增 provider 包 |
| 租户解析 | `kit-go/tenant.Resolver` | `tenant/default` | 改配置 |
| 事件总线 | `kit-go/event.Bus` | `event/nats` | 改配置 + 新增 provider 包 |
| 数据库 | `kit-go/db.DB` | `db/postgres` | 改 DSN + 重写 SQL（如换 MySQL） |
| 日志 | `kit-go/log.Logger` | `log/zap` | 改配置 |
| 配置中心 | `kit-go/config.Loader` | `config/env` | 改配置 |
| 服务发现 | `kit-go/discovery.Registry` | `discovery/k8s` / `discovery/static` | 改配置 |
| 部署 | 不侵入代码 | K8s Helm / Docker Compose | 纯运维切换 |

> **重要**：业务代码中不出现任何具体基础设施的 import（如 `github.com/open-policy-agent/...`、`github.com/nats-io/...`），全部通过 Kit 抽象接口使用。

## 十一、标准开发工作流

```text
1. 业务 PM 提需求
        ↓
2. 平台/架构师评审，确认领域边界
        ↓
3. 业务开发创建服务：
        $ kit new service order
        （生成：api/、internal/、migrations/、web/、Dockerfile）
        ↓
4. 编写业务 proto：
        $ vim services/order/api/order.proto
        ↓
5. 生成代码：
        $ kit generate
        （生成：pb.go、gateway、sqlc 模板、{service}-sdk-api）
        ↓
6. 编写 SQL 文件，运行 sqlc generate
        ↓
7. 编写领域逻辑（domain / app / repo）
        ↓
8. 编写前端子应用页面（使用 kit-ui + {service}-sdk-kit）
        ↓
9. 本地启动：
        $ kit dev
        （自动拉起：PostgreSQL、NATS、MaxKey、OPA、Gateway、本服务）
        ↓
10. 提交 PR → CI 构建镜像 → 部署到测试/生产
```

## 十四、关键决策记录

| 部署形态 | **K8s + Docker Compose + 二进制** | 覆盖云原生与私有化交付 |
